설치·디버그·모델전환·생성·합성·조립까지 전부 CLI로 굴렸고, 결과물은 캐릭터 로스터 + 총출동 4컷 만화. 다만 가는 길에 삽질을 꽤 했습니다. 그 과정을 처음부터 끝까지 적었어요.

개발·AI 얘기를 SNS에 올리다 보면 글만으론 좀 심심하더라고요. 캐릭터가 나오는 4컷 만화 한 장이면 훨씬 잘 읽히겠다 싶었어요. 그래서 이틀에 걸쳐 로컬 AI로 캐릭터 만화 뽑는 파이프라인을 직접 세워봤습니다. 클라우드 API 안 쓰고, 제 맥북 한 대에서요.

01

조건을 두 개 걸었어요

시작할 때 스스로 조건을 두 개 걸었어요. 이게 도구 선택을 거의 다 결정했습니다.

01

캐릭터 일관성

4컷이면 같은 캐릭터가 네 번 나와야 해요. 컷마다 딴 사람이 되면 만화가 아니죠. 이게 이미지 생성에서 제일 어려운 부분입니다.

02

CLI로, 에이전트가 직접

저는 마크다운 기반으로 작업하고 AI 코딩 에이전트로 자동화해요. 그래서 클릭해서 쓰는 GUI 툴(Draw Things, ComfyUI)은 후보에서 빠졌어요. "대본 쓰면 → 에이전트가 알아서 이미지까지"가 목표였습니다.

이 두 번째 조건 덕분에 선택지가 확 줄었습니다.

02

도구: mflux (MLX 네이티브 CLI)

애플 실리콘에서 로컬 이미지 생성을 하려면 MLX(애플의 머신러닝 프레임워크)를 타는 게 제일 빠릅니다. 그걸 CLI로 감싼 게 mflux예요. Huggingface Diffusers를 MLX로 포팅한 프로젝트인데, 재밌는 게 공식 문서에서 "CLI 탐색은 코딩 에이전트로 하라"고 대놓고 권합니다. 제 두 번째 조건이랑 딱 맞았어요.

pip install mflux 한 줄로 깔리고, 모델마다 전용 명령이 따로 있습니다(mflux-generate-flux2, mflux-generate-flux2-edit 이런 식). 이 "모델별 전용 명령" 구조를 처음에 몰라서 한 번 넘어졌는데, 그 얘기는 아래에서요.

참고로 제 머신은 맥북 프로 M3 Pro · GPU 14코어 · 통합 메모리 36GB입니다. 이 사양이 이 글 내내 중요하게 작용해요.

03

첫 삽질: 무거운 모델은 M3 Pro에서 죽습니다

처음엔 욕심을 부렸어요. 최신이라길래 z-image-turbo(6B)부터 붙였습니다. 결과는 삼연속 실패였어요.

❌ z-image-turbo 6B
명령 구조범용 플래그 → vae 못 찾고 죽음
모델 크기31 GB
생성 속도512²에서도 3~5분 무응답
GPU 워치독Timeout — 강제 종료
M3 Pro 36GB엔 과부하
✅ flux2-klein-4b 4B
명령 구조모델 전용 명령 사용
모델 크기15 GB
생성 속도14~22초 / 장 (4step)
GPU 워치독안 걸림 · 피크 10.5GB
스위트스팟

z-image가 삼연속으로 넘어진 이유는 세 가지였어요.

명령이 틀렸다 — mflux-generate --base-model z-image-turbo 같은 범용 플래그로 부르니 vae 컴포넌트를 못 찾는다고 죽더라고요. 알고 보니 모델마다 전용 명령을 써야 했습니다(mflux-generate-z-image-turbo).

모델이 31GB — 6B인데 텍스트 인코더·VAE까지 풀로 받으니 다운로드만 한참이었어요.

Metal GPU 워치독에 걸려 죽었다 — 이게 핵심입니다. 1024 해상도로 돌리니 첫 스텝에서 Command buffer execution failed ... Timeout. macOS는 GPU 커맨드 하나가 너무 오래 걸리면 강제로 죽이는데(워치독), 무거운 모델이 딱 거기 걸렸어요. 해상도를 512로 낮춰도 3~5분 안에 그림 한 장을 못 뽑았습니다.

교훈 하나: 파라미터가 크다고 좋은 게 아니라, 내 하드웨어에 맞아야 합니다. 6B는 M3 Pro 36GB엔 과했어요.

04

해결: 4B가 스위트스팟이었어요

무게를 낮췄습니다. flux2-klein-4b(4B)로 갈아탔더니 딴 세상이었어요. 다운로드 15GB(절반 수준), 생성 한 장에 14~22초(4스텝), 피크 메모리 10.5GB로 36GB에 여유가 넉넉했고, GPU 워치독은 안 걸립니다.

같은 맥북인데 모델 하나 바꿨다고 "안 됨"에서 "잘 됨"으로 넘어갔어요. 모델 무게가 로컬 추론의 진짜 제약이라는 걸 몸으로 배웠습니다.

05

진짜 난제들은 여기서부터였어요

모델이 돈다고 만화가 되는 건 아니었어요. 실제로 쓸만한 캐릭터 만화를 뽑으려니 문제가 줄줄이 나왔고, 하나씩 풀었습니다.

01 캐릭터 일관성 = 이미지→이미지 편집

텍스트로만 뽑으면 매번 딴 캐릭터가 나와요. 대신 flux2-edit(image-to-image)에 기준 이미지를 물려주고 "같은 캐릭터, 이번엔 이런 포즈"라고 시키면 얼굴·옷·소품이 유지됩니다. 4컷의 제일 큰 난제가 여기서 풀렸어요.

02 표정 바꾸면 선글라스가 벗겨진다

제 캐릭터들이 선글라스를 꼈는데, "놀란 표정" "눈썹 올리고" 같이 눈을 언급하면 모델이 눈을 그리려고 선글라스를 지워버리더라고요. 해법은 단순했어요. 눈 얘기는 빼고, 입·손·고갯짓만 바꾸라고 시키니 선글라스가 살았습니다.

03 한글은 모델 밖에서

로컬 diffusion 모델은 한글 렌더가 약해요. 그래서 말풍선·자막은 모델이 아니라 HTML로 합성했습니다. 그림은 AI가, 글자는 HTML→스크린샷으로. 저는 어차피 마크다운·HTML로 작업하니 워크플로에 딱 맞았어요.

04 CRT 스캔라인은 사실 실수에서 나왔어요

시각화 페이지에 가로 주사선(CRT 느낌) 오버레이를 깔아뒀는데, 그게 캐릭터 위에 겹쳐 보이는 게 예뻤어요. 그래서 아예 그 가로줄을 이미지에 구워넣었습니다(픽셀 행마다 살짝 어둡게). 레트로 게임 화면 같은 통일감이 생겼죠. (지금 이 페이지에도 그 스캔라인이 깔려 있어요.)

05 두 캐릭터를 한 컷에 = 배경 투명화 후 합성

대화 컷은 두 명이 마주봐야 하는데, 각 캐릭터는 주황 배경 위에 따로 뽑혀요. 그래서 주황 배경을 투명하게 도려내고(크로마키) 공유 씬 위에 나란히 올렸습니다. 함정이 있었는데, 스캔라인 명암차 때문에 단순 색 임계값으론 배경이 안 지워졌어요. 가장자리에서부터 채워 들어가는 방식(flood fill)으로 바꿔 해결했습니다.

06 키 맞추기

캐릭터마다 뽑히는 키가 제각각이라 나란히 세우면 들쭉날쭉했어요. 각 캐릭터의 실제 높이를 재서 같은 높이로 정규화하고 발끝을 맞췄습니다.

06

그래서 뭐가 나왔냐면

이 파이프라인으로 캐릭터 로스터를 만들었어요. 메인 둘(빨간 머리·금발), 엑스트라 셋(각자 성격 있음). 캐릭터마다 표정·방향·"노트북 하는 모습"·"먹는 모습" 같은 변형을 8종씩 미리 뽑아뒀습니다. 이걸 정본 JSON 한 파일로 관리해요. 새 만화 만들 때 재생성 없이 꽂아 쓰려고요.

메인 2(알통·오토토) + 엑스트라 3(9222·2329·0264) · 캐릭터마다 표정·포즈 변형 8종 · 정본 JSON 한 파일 관리

그리고 오늘, 그 캐릭터들 총출동 4컷을 완성했습니다. 주제는 제가 늘 밀던 거예요 — "AI가 코드 다 짜준 날, 팀의 반응." 누구는 "완벽해요, 다 봤어요"(사실 안 봄), 누구는 시킨 대로 통과만 누르고, 누구는 "돌아가던데요" 하고 넘기는데, 딱 한 명이 한 줄씩 읽다가 버그를 잡아냅니다.

maker ≠ checker — AI가 짜도, 읽고 이해하는 사람이 검증하는 거예요.

07

정리하면

  • 01로컬 AI로 만화 파이프라인, 됩니다. 설치·디버그·모델전환·생성·합성·조립까지 전부 CLI 자동화로 굴렸어요. 클라우드 크레딧 0원.
  • 02모델 무게가 관건. 4B는 M3 Pro에서 잘 돌고, 6B는 워치독에 죽어요. 큰 모델이 항상 답은 아닙니다.
  • 03일관성은 image-to-image, 한글·조립은 모델 밖에서(HTML). 각자 잘하는 걸 시키니 깔끔했어요.
  • 04제일 좋았던 건, 실수(스캔라인)가 스타일이 되고 삽질(z-image)이 교훈이 된 거예요. 원래 만드는 게 다 그렇더라고요.

혹시 애플 실리콘에서 로컬 이미지 생성 붙여보신 분? 어떤 모델 쓰시는지 궁금해요. 4B가 제 스위트스팟이었는데, 다른 분들 경험도 듣고 싶습니다.
