빅테크 디자인 시스템 비교 2026
— Material, Carbon, 그리고 Meta Astryx

토큰·컴포넌트·접근성으로 수렴한 판에서 새 차별축이 된 AI/MCP 대응, 그리고 그 첫 대형 실물인 Meta Astryx를 정리했어요.

tags: 디자인시스템 · 빅테크 · Astryx · MCP · StyleX · 프론트엔드

세 줄 요약 ↓

  1. 빅테크 디자인 시스템은 토큰 + 컴포넌트 + 접근성이라는 같은 골격으로 수렴을 끝냈어요. 표로 놓고 보면 정말 다 비슷합니다.
  2. 이제 갈리는 축은 하나, AI/MCP 대응이에요. 디자인 시스템을 사람만 읽는 게 아니라 AI가 직접 읽고 쓰게 여느냐.
  3. Meta가 공개한 Astryx가 그걸 전면에 내건 첫 대형 사례예요. 내부에서 8년, 13,000개 넘는 앱을 굴리던 시스템을 MIT로 풀면서 MCP를 같이 들고 나왔습니다.

01

진짜 뉴스는 오픈소스가 아니에요

Meta가 Astryx를 공개했다는 소식을 처음 봤을 때는 "또 하나 나왔네" 정도였어요. 빅테크가 디자인 시스템을 오픈소스로 푸는 건 이제 뉴스가 아니거든요. Google, IBM, Shopify, Atlassian까지 다 하고 있으니까요.

근데 내용을 뜯어보니 헤드라인이 잘못 잡혔다는 생각이 들었어요. 내부에서 8년을 키우고 13,000개 넘는 앱을 굴리던 시스템을 MIT 라이선스로 풀었다, 여기까지는 익숙한 이야기예요. 진짜 뉴스는 그 옆에 붙어 있던 한 줄이었습니다. 프로젝트 스캐폴딩과 CLI를 MCP로 노출한다는 것.

MCP(Model Context Protocol) — AI 도구가 외부 시스템을 읽고 쓸 수 있게 연결하는 규약이에요. 그러니까 Astryx는 사람 개발자용 라이브러리로 끝나는 게 아니라, AI 코딩 도구가 디자인 시스템을 직접 읽고 프로젝트를 깔아주는 경로까지 공식 기능으로 내놓은 거예요.

이게 왜 큰일인지는 판 전체를 놓고 보면 더 잘 보여요.

7행 × 6열 — Astryx 행만 강조 표시

표를 보면 두 가지가 동시에 보여요. 정말 다 비슷하다는 것. 그리고 Astryx 줄에만 다른 종류의 단어가 들어 있다는 것.

03

표에서 읽히는 패턴 넷

패턴 1

골격은 수렴이 끝났어요

어느 시스템을 봐도 토큰(색·타이포·간격 같은 디자인 값) 위에 컴포넌트를 쌓고, 접근성을 기본으로 까는 구조예요. Foundation에서 Component, Pattern으로 올라가는 계단은 이제 업계 공통 문법입니다.

패턴 2

오픈소스가 기본값이에요

표에서 유일한 예외가 Apple HIG인데, 이건 코드 라이브러리가 아니라 가이드 문서 중심이라 결이 달라요. 나머지는 전부 MIT나 Apache 같은 라이선스로 열려 있어요.

패턴 3

토큰은 표준으로 가는 중이에요

'디자인 토큰'이라는 개념 자체가 Salesforce Lightning에서 나왔고, 지금은 DTCG(Design Tokens Community Group)에서 도구를 가리지 않고 통하는 표준 포맷으로 다듬고 있어요. 어느 표를 봐도 토큰 열이 비어 있지 않은 이유예요.

패턴 4

그리고 AI 대응이 새 전선이에요

앞의 셋이 "왜 다 비슷해졌나"에 대한 답이라면, 이건 지금 막 갈리기 시작한 축이에요. 아래에서 따로 봅니다.

04

Astryx를 조금 더 뜯어보면

스택
React + StyleX
StyleX는 Meta가 만든 아토믹 CSS 엔진인데, 스타일을 빌드 시점에 컴파일하는 방식이에요. 여기에 CSS 커스텀 속성 기반 디자인 토큰을 얹어서 다중 테마를 지원합니다.
컴포넌트 수
소스마다 상이
마케팅 페이지는 160+, 문서는 150+, 일부 커버리지 자료는 90+로 표기가 갈려요. 정확한 숫자 하나를 박기는 어렵고, 150개 안팎의 접근성 지원 React 컴포넌트로 잡는 게 정직한 것 같아요.
구조
3계층 + 템플릿
Foundations — 타이포·색·레이아웃 토큰
Components — UI 빌딩 블록
Patterns — 테이블·폼·내비게이션처럼 검증된 설계 묶음
+ 프로덕션 템플릿 — 바로 쓸 수 있는 완성 골격
개방성
잠그지 않음
컴포넌트 소스를 내 프로젝트로 추출해서 고치는 swizzle을 공식 지원하고, Tailwind · CSS modules · 일반 CSS와도 호환돼요. 규칙으로 강제하기보다 문서와 예제로 안내하는 guide-first를 표방하고요.
MCP + CLI
프로젝트 스캐폴딩을 MCP로 노출해서, AI 에이전트가 Astryx 기반 프로젝트를 직접 세팅하고 컴포넌트를 쓰게 했어요. 저는 이 마지막 항목이 나머지 전부와 급이 다른 이야기라고 봐요.
05

AI/MCP 축, 지금 판세

MCP 네이티브공식 확인 참고회사 DS 아닌 툴 계층 미확인공개 자료에서 확인 못 함 — 단정 아님
시스템AI/MCP 대응
Astryx (Meta)MCP 네이티브스캐폴딩·CLI를 MCP로 노출, 코드 라이브러리 자체가 AI에 열림
(참고) Figma Dev Mode MCP참고디자인 소스 쪽을 AI에 여는 계열 — 회사 DS가 아니라 툴 계층
(참고) Vercel v0참고레지스트리·툴 계열의 AI 네이티브 접근
Material · Carbon · SLDS · Polaris · Atlassian미확인네이티브 MCP 지원을 공개 자료에서 확인 못 함

주의점 하나. 아래 다섯이 "안 한다"는 뜻이 아니에요. 제가 공개 자료에서 확인하지 못했다는 것까지가 사실이고, 내부에서 준비 중일 수도 있어요. 다만 적어도 공개된 범위에선, 빅테크가 자기 디자인 시스템에 AI 대응을 전면으로 내건 대형 사례는 Astryx가 처음이에요.

방향 차이도 재미있어요. Figma의 MCP가 디자인 파일을 AI가 읽게 하는 쪽이라면, Astryx는 코드 라이브러리를 AI가 쓰게 하는 쪽이에요. 같은 MCP인데 서 있는 자리가 달라요. 디자인 소스에서 출발하느냐, 코드에서 출발하느냐.

06

남은 생각

디자인 시스템은 오랫동안 "사람이 읽는 라이브러리"였어요. 문서 사이트 열어두고, 컴포넌트 이름 찾아서, 사람이 조립하는 것. 근데 코드를 조립하는 주체에 AI가 섞이기 시작하면서, 디자인 시스템의 정의가 "AI가 소비하는 컨텍스트 인프라" 쪽으로 밀리고 있는 것 같아요. Astryx는 그 방향에 놓인 첫 큰 돌이고요.

디자인 시스템은 "사람이 읽는 라이브러리"에서 "AI가 소비하는 컨텍스트 인프라"로 밀리는 중.

오늘 딱 하나 해볼 만한 게 있다면 이거예요. 지금 쓰는 AI 코딩 도구에 "우리 프로젝트 디자인 시스템의 버튼 규칙 알아?"라고 물어보는 것. 30초면 됩니다. 술술 답하면 이미 AI가 읽을 수 있는 형태로 정리돼 있는 거고, 얼버무리면 그 간극이 위에서 말한 새 전선이에요.

여러분 팀의 디자인 시스템은, AI가 읽을 수 있나요?
